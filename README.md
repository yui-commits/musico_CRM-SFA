## プロジェクト概要

音楽教室「ムジコ」の新規開拓営業を支援する **営業支援システム（CRM/SFA）** のリポジトリです。  
幼稚園・保育園向けバイオリン教室の **リード獲得（テレアポ）〜商談・体験会〜開講判断〜データ分析** までを一元管理し、  
営業効率と成約率の最大化を目指します。

- **対象ドメイン**: 幼稚園・保育園向け課外バイオリン教室の新規開拓営業
- **主な利用者**: テレアポ担当者、営業担当者、マネージャー・経営層
- **システム種別**: 社内向けWebアプリ（CRM/SFA＋ダッシュボード）

詳細な要件は `要件定義/` 配下のMarkdownに整理されています。

---

## システム全体像

本システムは、以下の3つの大きな機能ブロックで構成されます。

1. **Pre-APO: テレアポ管理（フェーズ1）**  
   - 未着手〜営業中〜NG までの **架電リスト管理と活動履歴の記録** を行う画面群  
   - AIが園の特色に合わせて生成する **専用営業トーク** を読みながら架電できるUI  
   - 「アポ獲得」時には Post-APO へ自動で案件が引き継がれる

2. **Post-APO: 商談・体験会進行管理（フェーズ2）**  
   - Trello 風の **カンバンボードUI** による商談〜体験会〜開講判断までの進行管理  
   - 面談日時、体験会日程、講師アサイン、チラシ送付、入会数などを1画面で管理  
   - ドラッグ＆ドロップでフェーズを進めると `deal_status` が更新される

3. **データ分析・インサイトダッシュボード（追加要件）**  
   - 期間×担当者で絞り込める **ダブル・フィルタリング** による分析画面  
   - 架電数、アポ率、開講率、エリア別・NG理由別の傾向などを可視化  
   - 営業ファネル、エリア別比較、アウトリーチ別開講率などのグラフを提供

これらは共通のデータモデル（施設マスタ＋活動履歴）を前提としており、  
**「1つの施設が、Pre-APO → Post-APO → 分析」へシームレスに流れる設計** になっています。

## データモデルの基本方針

システムの要件は、`要件定義/【全体概要】MUSICO_CRM・SFA要件定義.md` に詳細がありますが、  
README では開発時に特に重要となるポイントのみを抜粋します。

- **施設マスタ (`facilities`)**
  - 幼稚園・保育園など施設単位の情報を管理
  - 代表的なカラム:
    - `phone_number`: 正規化済み電話番号（ユニークキー、CSVアップロード時のマッチングキー）
    - `lead_status`: Pre-APO 用のリードステータス（未着手／営業中／NG／アポ獲得 等）
    - `deal_status`: Post-APO 用の商談フェーズ（面談日程調整中〜開講決定 等）
    - `features`: 園の特色・方針（AIトーク生成のインプットになるテキスト）
    - `ai_sales_script`: 園別に生成されたAI営業トーク（任意編集可）

- **活動履歴 (`activities` / `call_logs`)**
  - 架電・メール・フォローなどの **時系列のアクションログ** を1対多で保持
  - 代表的なカラム:
    - `sales_person`: 架電担当者名（LocalStorageから自動セット）
    - `called_at`: 架電日時（ダッシュボードの期間集計の基準）
    - `call_status`: 架電ステータス（未着手／不在／資料送付／アポ獲得／NG 等）
    - `next_action` / `next_action_date`: 次に何をいつやるか

この2テーブルを軸に、Pre-APO・Post-APO・ダッシュボードがそれぞれ異なる切り口でデータを利用します。

## 要件定義ドキュメントの構成

実装時に参照するべき主要な要件定義ファイルは以下です。

- **全体概要・データ設計**
  - `要件定義/【全体概要】MUSICO_CRM・SFA要件定義.md`
  - システム全体の目的、DBテーブル定義、CSV一括登録仕様、AIトーク生成、簡易認証などの中核要件

- **フェーズ1: テレアポ管理（Pre-APO）**
  - `要件定義/【フェーズ1】テレアポ管理画面（Pre-APO）機能・UIUX要件定義.md`
  - ダッシュボード兼リスト画面、施設詳細＋架電画面、アポ獲得時の自動遷移などのUI/UX要件

- **フェーズ2: 商談・体験会進行（Post-APO）**
  - `要件定義/【フェーズ2】商談管理・体験会進行画面（Post-APO）機能・UIUX要件定義.md`
  - カンバンレーン構成、施設カード表示内容、進捗入力パネル、D&D連動ロジックなど

- **追加要件: データ分析ダッシュボード**
  - `要件定義/【追加要件】データ分析・インサイトダッシュボード機能 詳細設計.md`
  - 期間・担当者フィルター、KPIサマリ、リーダーボード、各種グラフの要件

まずはこれら4ファイルを読めば、システムの全体像と画面要件を把握できます。

## 今後の実装フェーズの進め方（推奨）

実装技術スタックはまだ固定していない前提で、フェーズ別の進め方を整理します。

- **フェーズ0: プロトタイピング / 設計**
  - UIワイヤーフレームを Figma 等で軽く起こし、要件定義とのギャップを確認
  - DBスキーマ（`facilities` / `activities`）をマイグレーションツールで定義

- **フェーズ1: Pre-APO画面の実装**
  - 施設一覧＋フィルタリング＋本日のタスクビュー
  - 施設詳細＋AIトーク表示＋活動履歴入力フォーム
  - `lead_status` と `call_status` の連動ロジック

- **フェーズ2: Post-APOカンバンの実装**
  - カンバンレーンとD&Dでの `deal_status` 更新
  - 詳細パネルでの体験会・講師・チラシ・入会数の管理

- **フェーズ3: ダッシュボードの実装**
  - 期間＆担当者フィルター
  - KPIサマリパネル、リーダーボード、主要グラフ

各フェーズの詳細仕様は `要件定義/` をソース・オブ・トゥルースとして随時更新していきます。
